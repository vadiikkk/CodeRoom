# CodeRoom

Открытая LMS-платформа для курсов по программированию: публикация материалов, задания с ручной и автоматической проверкой (включая код на **Go / Python / Java**), оценки и ведомость. Репозиторий содержит **клиент** (веб-SPA), **серверную часть** (микросервисы на Kotlin) и **контур автопроверки** (Kafka → worker с запуском тестов в изоляции).

## Основные возможности

- **Роли:** глобально - студент, ассистент, преподаватель, root; внутри курса - студент / ассистент / преподаватель курса.
- **Курсы:** структура (блоки, порядок, видимость), материалы (текст и вложения), регистрация участников, группы для групповых заданий.
- **Задания:** текстовые, с загрузкой файлов, **кодовые** с шаблонным репозиторием GitHub, PR-ами и автопроверкой; дедлайны, вес в оценке, индивидуальная / групповая работа.
- **Проверка и оценки:** ручная оценка и комментарии; для кода - прогон тестов и выставление баллов по правилам; журналы для преподавателя/ассистента и личный просмотр для студента.
- **Файлы:** загрузка и скачивание через **presigned URL** в объектное хранилище (S3-совместимое, в стенде - MinIO).
- **Идентификация:** JWT (access/refresh), администрирование пользователей на уровне системы (root).

## Архитектура (типовой стенд)

- **`api-gateway`** - единая точка входа для HTTP: CORS, лимиты, проверка JWT, маршрутизация к сервисам, общий Swagger UI.
- **`identity-service`** - регистрация, логин, refresh, сессии, глобальные роли, root-admin.
- **`course-service`** - домен LMS: курсы, структура, материалы, задания, GitHub PAT курса, сдачи, оценки, gradebook.
- **`content-service`** - подпись URL для загрузки/скачивания объектов в хранилище.
- **`autograding-service`** - оркестрация заданий автопроверки через Kafka.
- **`runner-service`** - исполнитель: клонирование PR, подмешивание приватных тестов, запуск команды из `.coderoom.yml`.

Инфраструктура на стенде: **PostgreSQL** (отдельные БД под сервисы), **Kafka**, **MinIO**. Клиент обращается к API по HTTP через gateway; долгие операции автопроверки - асинхронно.

## Конфигурация кодовых заданий (`.coderoom.yml`)

В шаблонный репозиторий задания кладётся файл **`.coderoom.yml`**. При сдаче решения (создании code attempt) платформа **читает актуальный файл из default branch репозитория задания**, валидирует его и сохраняет **снапшот** YAML вместе с попыткой - дальше автопроверка идёт по снапшоту. Раннер клонирует PR, при необходимости распаковывает zip с приватными тестами в `privateTests.targetPath`, затем запускает **`check.command`** в каталоге **`check.workdir`** (относительно корня репозитория).

Парсер строгий: **неизвестные ключи в YAML запрещены** (`FAIL_ON_UNKNOWN_PROPERTIES`). Имена перечислений в файле - как в API: **`GO`**, **`PYTHON`**, **`JAVA`**, **`UNIT_TESTS`**, **`SCRIPT`**, **`ALL_OR_NOTHING`**, **`PASS_RATE`**.

### Поддерживаемые поля

| Поле | Тип / значения | Смысл и ограничения |
|------|----------------|---------------------|
| `version` | целое | Сейчас поддерживается только **`1`**. |
| `language` | `GO` \| `PYTHON` \| `JAVA` | Должен **совпадать** с языком code-задания на платформе (иначе сдача отклоняется). |
| `check.type` | `UNIT_TESTS` \| `SCRIPT` | Для раннера оба варианта ведут себя одинаково: выполняется одна shell-команда; тип отражает **семантику** задания (тесты vs произвольный скрипт). |
| `check.command` | строка | Обязательна, не пустая. Запуск через `sh -lc` в Linux-контейнере раннера (см. образ `runner-service`). |
| `check.workdir` | строка, по умолчанию `.` | Относительный путь от корня репозитория: **без** ведущего `/` и **без** `..` (защита от выхода за пределы репо). |
| `attempts.max` | целое > 0 | Логический лимит в конфиге репозитория; фактический лимит попыток на платформе задаётся при создании задания - учитывайте оба. |
| `privateTests.enabled` | boolean, по умолчанию `false` | Если `true`, на этапе прогона раннер скачивает zip приватных тестов, привязанный к заданию на платформе, и распаковывает его. |
| `privateTests.targetPath` | строка, по умолчанию `.` | Каталог **внутри репозитория**, куда кладётся содержимое zip (относительный путь; при `enabled: true` не может быть пустым). Рекомендуется явно задавать путь, чтобы не дублировать имена модулей с публичными тестами (см. ниже). |
| `scoring.strategy` | `ALL_OR_NOTHING` \| `PASS_RATE` | **`ALL_OR_NOTHING`**: максимум баллов при коде выхода `0`, иначе 0. **`PASS_RATE`**: доля пройденных тестов × `maxScore` - для **Go** считается по выводу `go test -json`, для **Python/Java** - по **JUnit XML** (раннер обходит репозиторий и суммирует корневые элементы `testsuite` / отчёты с атрибутами `tests`, `failures`, …). Если разобрать тесты не удалось, используется запасной вариант по коду выхода команды (как all-or-nothing). |
| `scoring.maxScore` | число | Строго **> 0**, в ответах API обычно с двумя знаками после запятой. |

### Пример (согласованный с парсером)

```yaml
version: 1
language: GO

check:
  type: UNIT_TESTS
  command: go test -json ./...
  workdir: "."

attempts:
  max: 3

privateTests:
  enabled: true
  targetPath: "."

scoring:
  strategy: PASS_RATE
  maxScore: 100.00
```

Для **Python** и **Java** при генерации шаблона в репозитории используются типовые команды (их можно заменить, сохранив совместимость со стратегией оценивания):

- **Python:** `python -m pytest --junitxml=junit.xml` - итоговый XML должен оказаться под деревом репозитория (раннер ищет любые `*.xml`).
- **Java:** `./gradlew test` - ожидаются отчёты Gradle/Surefire в виде JUnit XML внутри репозитория.

### `privateTests.targetPath` и шаблон из `course-service`

Если в YAML не указать `targetPath`, при **создании** code-задания на платформе в сгенерированный `.coderoom.yml` подставляются языковые значения по умолчанию:

| Язык | Дефолт `targetPath` |
|------|---------------------|
| `GO` | `.` |
| `PYTHON` | `tests` |
| `JAVA` | `src/test/java` |

Zip с приватными тестами должен содержать **файлы с корректными относительными путями** (без zip-slip); при `targetPath: "."` путь в compose/раннере нормализуется, чтобы проверка безопасности не ломалась.
